XSS Testing Lab

A deliberately vulnerable sandbox. Every input is reflected without sanitization.

⚠️ This page is intentionally vulnerable — for security research and education only. Never deploy on production.

Defense Mode

Toggle between vulnerable behavior and safe handling to compare exploitability vs mitigation.

Output Encoding (HTML/DOM)

Untrusted input is rendered with textContent instead of innerHTML, so tags/events are displayed as text and cannot execute.

Safe Attribute Handling

Attributes are assigned with setAttribute() on created elements, preventing quote-breakout payloads from injecting new handlers.

No Dynamic Code Execution

User input is treated as data only. Defense mode removes the dangerous eval path in JS context examples.

Input Treated as Content

Interaction-zone payloads are printed as text rather than parsed as HTML, blocking event-driven payload execution.

Reflected XSS

User input is placed directly into innerHTML with no sanitization. Classic server-side reflection, simulated client-side.

// Vulnerable server-side code (PHP equivalent):
echo "Hello " . $_GET['name'];  // raw reflection

// Vulnerable client-side simulation:
element.innerHTML = userInput;  // ← no sanitization
Try: <img src=x onerror=alert(1)> or <svg onload=alert(1)>

The payload comes from location.hash — the URL fragment. The server never sees this data; it's purely client-side.

  decodeURIComponent(location.hash.slice(1));  // ← sink

Attribute XSS

Input is placed inside an HTML attribute value. Attack requires breaking out of the attribute with a quote.

// Vulnerable template:
<div title="USER_INPUT">hover me</div>
// If USER_INPUT = " onmouseover="alert(1)
// Result: <div title="" onmouseover="alert(1)">
Try: " autofocus onfocus="alert(1)" or " onmouseover="alert(1)
